# AI 协作开发参考提示词

> 按需求规模分层使用。不是模板，不需要每次照搬，根据实际情况调整措辞。

---

## 一、大需求 → 需求设计文档

适用于：跨多个模块、预计超过一周的大需求。产出一份需求设计文档，含 Epic 拆解。

### 1.1 启动讨论

```
我有一个需求想和你讨论。先不写代码，帮我理清思路。

{用自然语言描述需求背景、目标用户、期望效果}

请：
1. 用你的话复述你理解的核心问题
2. 列出你需要我澄清的问题
3. 识别可能的风险和复杂点
```

> 回答 AI 的澄清问题，可能来回几轮。

### 1.2 拆解 Epic

```
理解一致了。现在帮我拆解为 Epic：
1. 每个 Epic 应该是独立可交付的
2. 标注 Epic 之间的依赖关系
3. 给出推荐的开发顺序
```

### 1.3 产出文档

```
把我们讨论的结论整理成一份需求设计文档，包含：
- 需求背景和目标
- 整体方案概述
- Epic 清单（含依赖关系和建议顺序）
- 我们讨论中确定的关键决策

存放到 {你指定的路径}
```

## 二、Epic / 中需求 → Backlog 文档

适用于：一个 Epic，或不需要上层设计文档的中等需求。产出 Backlog 文档，含 Feature 拆解。

### 2.1 启动讨论

```
我要开始做 {Epic/需求名称}。

{粘贴上层设计文档中该 Epic 的描述，或直接描述需求}

请先分析：
1. 需要哪些 Feature 来完成
2. Feature 之间的依赖
3. 有什么需要我澄清的
```

### 2.2 产出 Backlog

```
讨论清楚了，整理成 Backlog 文档：
- 每个 Feature 一段话描述目标和范围
- 标注依赖关系和建议顺序

存放到 {你指定的路径}
```

## 三、Feature / 小需求 → Superpowers 开发

适用于：Backlog 中的一个 Feature、独立的小需求、或 Bug 修复。这是最常用的流程。

### 3.1 需求澄清（brainstorming）

**完整 Feature，从头开始：**

```
我要开发这个 Feature：

{粘贴 Backlog 中的 Feature 描述，或直接描述需求}

先做需求分析，不要写代码。
```

> AI 会启动 brainstorming 技能。讨论中你可以随时打断、补充、纠正。
> brainstorming 结束后 AI 会给出总结，确认后再继续。

**需求已经很清楚，跳过讨论：**

```
需求很简单，不需要讨论，直接做计划：

{描述清楚要做什么、验收标准是什么}
```

**Bug 修复：**

```
有个 Bug 需要修复：

现象：{描述现象}
复现步骤：{如何复现}
期望行为：{应该是什么样}

{如果有报错日志，贴上}
```

> AI 会启动 systematic-debugging 技能。

### 3.2 制定计划（writing-plans）

**正常流程，从 brainstorming 衔接：**

```
分析到位了，开始制定实施计划。
```

> AI 会启动 writing-plans 技能，产出计划文件。

**你想控制计划粒度：**

```
制定实施计划，注意：
- 每步不超过 150 行变更
- 测试和实现分开为独立步骤
- 先列出计划让我确认，不要直接执行
```

**你已经知道怎么做，只需要 AI 执行：**

```
不需要做计划，按以下步骤执行：
1. {步骤 1}
2. {步骤 2}
3. ...

每步完成后停下来让我确认。
```

### 3.3 执行实现（TDD）

**按计划逐步执行：**

```
计划没问题，开始执行。每完成一步先跑测试，然后告诉我结果，等我确认后再继续下一步。
```

**让 AI 自主推进，只在关键点停：**

```
开始执行计划。常规步骤直接推进，遇到以下情况停下来跟我确认：
- 需要修改已有代码的设计
- 测试失败且不确定原因
- 发现计划有遗漏需要调整
```

**让 AI 全部做完：**

```
开始执行计划，全部做完后告诉我最终结果。
```

**从计划中间开始（上次中断了）：**

```
继续执行计划，从 Step {N} 开始。之前的步骤已经完成了。

{如果有上下文需要补充：}
上次做到 Step {N-1}，当时的情况是 {简要说明}。
```

**只做计划中的某几步：**

```
只执行计划中的 Step {X} 和 Step {Y}，其他跳过。
```

### 3.4 验证完成（verification）

**正常验收：**

```
实现完了，帮我做最终验证：
1. 跑全量测试
2. 检查是否有遗漏的场景
3. 确认代码符合项目规范
```

**需要跑变异测试（Phase 5 级别的审查）：**

```
Feature 完成，做完整审查：
1. mvn test（全量测试）
2. mvn org.pitest:pitest-maven:mutationCoverage -pl 模块名（变异测试，杀死率 >= 70%）
3. 检查架构规范是否通过
4. 有没有值得记录到 docs/PITFALLS.md 的踩坑经验
```

## 四、踩坑记录更新

在开发过程中遇到了花时间排查的问题，或 AI 反复犯的错误时触发。

```
这个问题值得记录。帮我追加到 docs/PITFALLS.md：
- 问题现象
- 根本原因
- 正确做法
- 记忆口诀（一句话总结）
```

> 不需要每个 Feature 结束都检查，遇到了就记，没遇到就不记。

---

## 五、归档

一个 Feature / Epic / 需求完成后，人工决定是否需要归档。

```
{Feature/Epic 名称} 已完成。帮我归档：

1. 从过程文档中提取有价值的内容，整理到 docs/guide/：
   - 这个模块/功能提供了什么能力
   - 核心 API 和使用示例
   - 重要的设计取舍
   - 注意事项
2. 检查 docs/PITFALLS.md 是否有需要补充的
3. 列出可以清理的过程文档

不要直接删除过程文档，列出来让我确认。
```

## 六、跨会话继续工作

新开一个会话，需要接续上次的工作。

**继续上次的 Feature：**

```
我在做 {Feature 名称}，上次完成到 {进度}。

计划文件在 {路径}，请读取后从 Step {N} 继续。
```

**接续上次的讨论：**

```
我们之前讨论了 {主题}，结论文档在 {路径}。

请读取后，我们继续下一步：{要做什么}。
```
