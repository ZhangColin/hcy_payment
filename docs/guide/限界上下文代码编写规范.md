# 第一部分：架构理念

## 一、六边形架构概述

### 1.1 架构层次与职责

每个限界上下文采用 DDD 六边形架构（端口适配器架构），清晰划分层次职责：

```
        ┌─────────────────────────────────────┐
        │       北向接口（Driving Side）        │
        │  REST API | GraphQL | gRPC | MQ     │
        └─────────────────────────────────────┘
                        ↓
        ┌─────────────────────────────────────┐
        │      应用层（Application Layer）      │
        │  北向接口适配层 | 上下文出入口         │
        └─────────────────────────────────────┘
                        ↓
        ┌─────────────────────────────────────┐
        │       领域层（Domain Layer）          │
        │  核心业务逻辑 | 南向端口接口          │
        └─────────────────────────────────────┘
                        ↓
        ┌─────────────────────────────────────┐
        │      南向接口（Driven Side）          │
        │  密码加密 | 持久化 | 缓存 | 外部服务  │
        └─────────────────────────────────────┘
                        ↓
        ┌─────────────────────────────────────┐
        │    基础设施层（Infrastructure）       │
        │  南向接口的适配器实现                 │
        └─────────────────────────────────────┘
```

**层次职责说明**：

| 层次 | 职责 | 典型组件 |
|------|------|---------|
| **北向接口** | 外部调用入口 | REST Controller、GraphQL Resolver、gRPC Service、MQ Listener |
| **应用层** | 北向接口适配层，上下文出入口 | AppService、DTO、Mapper |
| **领域层** | 核心业务逻辑，按职责定义接口 | 聚合根、实体、领域服务 |
| **南向接口** | 需要的外部能力 | Repository、端口接口（如密码加密器） |
| **基础设施层** | 南向接口的适配器实现 | JPA Repository、BCrypt 适配器、Redis 适配器 |

**关键原则**：

1. **应用层是上下文的出入口**：
   - 北向接口（REST、GraphQL 等）通过应用层访问领域逻辑
   - 其他上下文通过本上下文的应用层进行跨上下文调用
   - 应用层负责业务流程编排，不包含业务规则

2. **依赖方向**：
   - 北向接口 → 应用层 → 领域层
   - 领域层定义南向端口接口
   - 基础设施层实现南向端口接口

3. **南向接口适配器按技术拆分**：
   - `persistence/` - JPA Repository 实现
   - `redis/` - Redis 缓存实现
   - `messaging/` - 消息队列适配器
   - 其他技术按需创建子包
   - **单个适配器时**：可直接放在 `infrastructure/` 根目录下

### 1.2 端口适配器模式

**端口接口定义原则**：

- **领域层需要**：在领域层按职责放置接口
  - **组织方式 1**：使用 `domain/port/` 包（推荐，明确标识为端口接口）
  - **组织方式 2**：按职责放在 `domain/service/`（当接口与领域服务紧密相关时）
  - 适配器实现在基础设施层

- **应用层需要**：接口定义和适配器都在基础设施层
  - 应用层可以依赖基础设施层
  - 接口和适配器分开文件（同一包下）
  - 示例：跨上下文调用

**示例 1**：使用 `domain/port/` 组织（推荐）

```java
// domain/port/PasswordEncoder.java（领域层定义端口接口）
@Port(PortType.CLIENT)
public interface PasswordEncoder {
    String encode(String plainPassword);
    boolean matches(String plainPassword, String encodedPassword);
}

// infrastructure/BCryptPasswordEncoderAdapter.java（基础设施层实现）
@Adapter(PortType.CLIENT)
public class BCryptPasswordEncoderAdapter implements PasswordEncoder {
    private final BCryptPasswordEncoder encoder = new BCryptPasswordEncoder(10);

    @Override
    public String encode(String plainPassword) {
        return encoder.encode(plainPassword);
    }

    @Override
    public boolean matches(String plainPassword, String encodedPassword) {
        return encoder.matches(plainPassword, encodedPassword);
    }
}
```

**示例 2**：应用层跨上下文调用

```java
// infrastructure/AdminUserClient.java（基础设施层定义接口）
public interface AdminUserClient {
    boolean isAdminExists(Long userId);
}

// infrastructure/LocalAdminUserClientAdapter.java（基础设施层实现）
@Component
public class LocalAdminUserClientAdapter implements AdminUserClient {
    private final AdminUserQueryAppService adminUserQueryAppService;

    @Override
    public boolean isAdminExists(Long userId) {
        return adminUserQueryAppService.existsById(userId);
    }
}

// 未来拆分后：infrastructure/RemoteAdminUserClientAdapter.java
@Component
public class RemoteAdminUserClientAdapter implements AdminUserClient {
    // 改为 HTTP/gRPC 调用
}
```

### 1.3 依赖倒置原则

**核心原则**：
> 领域层必须保持零外部依赖。所有外部框架、库、基础设施都通过端口接口解耦。

**禁止事项**：
```java
// ❌ 禁止：领域层直接依赖 Spring Framework
import org.springframework.security.crypto.bcrypt.BCryptPasswordEncoder;

// ❌ 禁止：领域层直接依赖 Redis 客户端
import redis.clients.jedis.Jedis;

// ❌ 禁止：领域层直接依赖 HTTP 客户端
import org.springframework.web.client.RestTemplate;
```

**正确做法**：
```java
// ✅ 正确：通过端口接口解耦
@DomainService
public class PasswordEncoderService {
    private final PasswordEncoder encoder;  // 端口接口
}
```

## 二、DDD 设计原则

### 2.1 小聚合原则

**核心思想**：聚合应该保持小，只包含真正需要维护一致性的对象。

**原则要点**：

| 原则 | 说明 | 示例 |
|------|------|------|
| **1. 一个聚合根** | 每个聚合只有一个聚合根 | `AdminUser` 是聚合根，`AdminUserRole` 是实体 |
| **2. 保持小** | 聚合包含的对象尽可能少 | `AdminUser` 只管理 ID 集合，不加载完整的 `AdminRole` 对象 |
| **3. ID 引用** | 聚合间通过 ID 引用，而非对象引用 | `AdminUserRole` 存储 `roleId: Long`，而非 `AdminRole` 对象 |
| **4. 边界修改** | 修改聚合只能通过聚合根 | `adminUser.addRole(roleId)` 而非直接操作 `AdminUserRole` |
| **5. 事务一致性** | 聚合是事务一致性的边界 | 修改聚合内的所有对象在同一个事务中完成 |

**为什么保持小聚合**？

1. **性能优化**：减少数据库加载和锁竞争
2. **并发友好**：小聚合锁住的数据少，并发冲突概率低
3. **简单清晰**：职责单一，易于理解和维护
4. **一致性边界**：聚合是事务一致性的边界，保证聚合内对象的完整性

**应用服务与聚合的协作**：

应用服务方法可以涉及多个聚合，这是**允许且常见的**：
- 应用服务可注入多个 Repository，查询并验证相关聚合的存在性
- 每个聚合内部的一致性由聚合根负责保证

**示例**：`assignRoles()` 方法
```java
@Service
public class AdminUserManagementAppService {
    private final AdminUserRepository adminUserRepository;
    private final AdminRoleRepository adminRoleRepository;  // 可注入多个 Repository

    @Transactional
    public void assignRoles(Long userId, AssignRolesCommand command) {
        // 1. 查询并修改 AdminUser 聚合
        AdminUser adminUser = requirePresent(
                adminUserRepository.findById(userId),
                AdminMessage.ADMIN_NOT_FOUND
        );

        // 2. 应用服务批量验证其他聚合的存在（避免 N+1 查询）
        Set<Long> existingRoleIds = adminRoleRepository.findAllById(command.roleIds())
                .stream()
                .map(AdminRole::getId)
                .collect(Collectors.toSet());

        if (!CollUtil.containsAll(existingRoleIds, command.roleIds())) {
            throw new ApplicationException(AdminMessage.ROLE_NOT_FOUND);
        }

        // 3. 修改 AdminUser 聚合（只修改 ID 引用）
        adminUser.clearRoles();
        for (Long roleId : command.roleIds()) {
            adminUser.addRole(roleId);
        }

        adminUserRepository.save(adminUser);
    }
}
```

### 2.2 限界上下文边界

**上下文定义**：
- 限界上下文是 DDD 的战略设计工具
- 每个上下文有自己的领域模型、语言、边界
- 上下文之间通过应用事件或应用层接口通信

**上下文类型（SubDomain 枚举）**：

| 类型 | 说明 | 示例 |
|------|------|------|
| **CORE** | 核心域：业务核心竞争力的来源 | 电商平台的商品推荐算法 |
| **SUPPORTING** | 支撑域：业务流程必须但非核心竞争力的能力 | 用户认证、权限管理、消息通知 |
| **GENERIC** | 通用域：所有行业都一样的通用能力 | 日志记录、文件存储、邮件发送 |

### 2.3 包结构设计原则

**标准包结构**：

```
com.aieducenter.{context}
├── package-info.java           # 上下文说明（JavaDoc）
├── domain/                     # 领域层
│   ├── aggregate/              # 聚合根
│   ├── entity/                 # 实体
│   ├── repository/             # 仓储接口
│   ├── service/                # 领域服务（按需创建）
│   ├── port/                   # 端口接口（推荐，按需创建）
│   ├── error/                  # 领域错误定义（CodeMessage 枚举）
│   └── enums/                  # 领域枚举（独立文件）
├── application/                # 应用层
│   ├── dto/                    # 数据传输对象
│   │   ├── command/            # 命令 DTO（写操作）
│   │   ├── query/              # 查询 DTO（读操作）
│   │   └── response/           # 响应 DTO
│   ├── mapper/                 # 领域对象与 DTO 转换器
│   └── {Scenario}AppService.java    # 应用服务
├── infrastructure/             # 基础设施层（南向接口适配器）
│   ├── persistence/            # 持久化实现（按需）
│   ├── redis/                  # Redis 缓存实现（按需）
│   ├── messaging/              # 消息队列适配器（按需）
│   ├── {technology}/           # 其他技术适配器（按需）
│   └── *或单个适配器直接放根目录*
└── endpoints/                  # 北向接口适配器层
    ├── controller/             # REST API（内部）
    ├── api/                    # 外部 API
    ├── listener/               # MQ Listener
    └── scheduler/              # 定时任务
```

# 第二部分：层次实现规范

## 三、领域层

### 3.1 聚合根（Aggregate）

**位置**：`domain/aggregate`  
**命名规范**：`{Entity}.java`（无需后缀）

**核心注解**：
```java
@Entity
@Table(name = "sys_admin_users")
@Aggregate
public class AdminUser extends AuditableSoftDeletable implements AggregateRoot<AdminUser> {
    private static final String USERNAME_PATTERN = "^[a-zA-Z][a-zA-Z0-9_]{2,19}$";

    @Getter
    @Id
    @Column(name = "id", nullable = false, updatable = false)
    private Long id;

    @Getter
    // 无需 @Convert 或 @Enumerated 注解
    // AdminUserStatus.JpaConverter 会自动应用
    @Column(name = "status", nullable = false)
    private AdminUserStatus status;

    @OneToMany(cascade = CascadeType.ALL, orphanRemoval = true)
    @JoinColumn(name = "admin_id")
    private final Set<AdminUserRole> userRoles = CollUtil.newHashSet();

    @PrePersist
    void prePersist() {
        if (id == null) {
            this.id = TsidGenerator.newInstance().generate();
        }
    }

    // 业务方法...
    public void addRole(Long roleId) {
        userRoles.add(new AdminUserRole(this.id, roleId));
    }

    public void clearRoles() {
        userRoles.clear();
    }

    public Set<Long> getRoleIds() {
        return userRoles.stream()
                .map(AdminUserRole::getRoleId)
                .collect(Collectors.toSet());
    }
}
```

**聚合根规范要点**：

1. 继承 `AuditableSoftDeletable`（**建议**：自动提供审计字段 + 软删除）
2. 实现 `AggregateRoot<T>` 接口
3. 使用 `@Aggregate` 注解标记
4. 简单字段使用 `@Getter` + `@Setter` 注解（字段级）
5. 有业务逻辑的方法手写（构造函数、业务方法）
6. 构造函数负责不变量校验
7. `@PrePersist` 生成 TSID
8. **关联管理**：使用 `@OneToMany` + `orphanRemoval = true`，关联实体物理删除

**聚合根设计建议**：

| 设计决策 | 建议 | 原因 |
|---------|------|------|
| **继承基类** | 继承 `AuditableSoftDeletable` | 自动提供审计字段（createdAt/updatedAt）+ 软删除能力 |
| **软删除** | 推荐软删除 | 保留数据审计记录，支持数据恢复 |
| **ID 生成** | 使用 TSID | 分布式友好，时间排序 |
| **聚合大小** | 保持小聚合 | 减少锁竞争，提高并发性能 |

**什么时候不使用软删除？**
- 日志数据：量大且无需恢复
- 临时数据：会话信息、缓存数据
- 关联表实体：避免唯一索引冲突

### 3.2 实体（Entity）

**位置**：`domain/entity`  
**命名规范**：`{Parent}{Entity}.java` 或 `{Entity}.java`

**核心原则**：关联表实体使用**单一代理主键**，而非 JPA 复合主键。

**聚合根 vs 实体的继承原则**：

| 类型 | 继承基类 | 删除方式 | 原因 |
|------|---------|---------|------|
| **聚合根** | `AuditableSoftDeletable` | 软删除 | 独立生命周期，需保留审计记录 |
| **实体** | 无基类（仅实现接口） | 物理删除 | 依附聚合根，避免唯一索引冲突 |

**为什么实体不继承 AuditableSoftDeletable**？
1. **唯一索引冲突**：关联表有唯一约束（如 `admin_id + role_id`），软删除后重建关联会报错
2. **DDD 小聚合原则**：实体是聚合的一部分，生命周期由聚合根管理
3. **简化设计**：关联删除就是物理删除，无需保留审计记录

**核心规范**：
```java
@Entity
@Table(name = "sys_admin_user_roles", uniqueConstraints = {
    @UniqueConstraint(columnNames = {"admin_id", "role_id"})
})
@Getter
public class AdminUserRole implements DomainEntity<AdminUserRole, Long> {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @Column(name = "id", nullable = false, updatable = false)
    private Long id;

    @Column(name = "admin_id", nullable = false)
    private Long adminId;

    @Column(name = "role_id", nullable = false)
    private Long roleId;

    protected AdminUserRole() {
    }

    public AdminUserRole(Long adminId, Long roleId) {
        this.adminId = adminId;
        this.roleId = roleId;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (o == null || getClass() != o.getClass()) return false;
        AdminUserRole that = (AdminUserRole) o;
        return Objects.equals(adminId, that.adminId) && Objects.equals(roleId, that.roleId);
    }

    @Override
    public int hashCode() {
        return Objects.hash(adminId, roleId);
    }
}
```

**实体规范要点**：

1. **不继承** `AuditableSoftDeletable`（**强烈建议**：避免唯一索引与软删除冲突）
2. 实现 `DomainEntity<T, ID>` 接口
3. 使用单一代理主键（`@GeneratedValue(strategy = GenerationType.IDENTITY)`）
4. 业务唯一性通过 `@UniqueConstraint` 保证
5. 重写 `equals()` 和 `hashCode()` 基于业务键（非主键）
6. 简单字段使用 `@Getter` 注解
7. **物理删除**：聚合根删除关联时直接调用 `repository.delete()`

### 3.3 仓储接口（Repository）

**位置**：`domain/repository`  
**命名规范**：`{Resource}Repository.java`

```java
public interface AdminUserRepository extends BaseRepository<AdminUser, Long> {

    Optional<AdminUser> findByUsername(String username);

    boolean existsByUsername(String username);

    @Query("SELECT COUNT(ur) > 0 FROM AdminUserRole ur " +
           "WHERE ur.adminId = :adminId " +
           "AND EXISTS (SELECT 1 FROM AdminRole r WHERE r.id = ur.roleId AND r.code = :roleCode AND r.deleted = false)")
    boolean hasRole(@Param("adminId") Long adminId, @Param("roleCode") String roleCode);
}
```

**注意**：`@Query` 中的 JPQL 需要手动添加 `deleted = false` 条件（`@SQLRestriction` 对 JPQL 不生效）

### 3.4 领域服务（DomainService）

**位置**：`domain/service`  
**命名规范**：`{Scenario}Service.java` 或 `{DomainConcept}Service.java`

**何时使用**：
- 涉及多个聚合根的复杂领域逻辑
- 不属于任何单一聚合根的业务规则

**架构原则**：DomainService 不涉及 DTO

| 返回类型 | 说明 |
|---------|------|
| 领域模型（聚合根/实体） | ✅ 可以返回 |
| 基本类型（Long/void/boolean） | ✅ 可以返回 |
| DTO | ❌ 禁止涉及 |

```java
@DomainService
public class RolePermissionService {

    private final AdminRoleRepository roleRepository;
    private final PermissionScanner permissionScanner;

    public List<String> validatePermissionCodes(Set<String> permissionCodes) {
        Set<String> validCodes = permissionScanner.scanByScope("admin").stream()
                .map(Permission::code)
                .collect(Collectors.toSet());

        return permissionCodes.stream()
                .filter(code -> !validCodes.contains(code))
                .toList();
    }
}
```

**领域服务规范要点**：

1. 使用 `@DomainService` 注解
2. 构造函数注入
3. 不涉及 DTO 转换
4. 可以返回领域模型
5. 可以被 AppService 调用

### 3.5 领域错误定义

**位置**：`domain/error`  
**命名规范**：`{Context}Message.java`

```java
public enum AdminMessage implements CodeMessage {
```

### 3.6 枚举与常量

#### 3.6.1 枚举规范

**原则**：
- 枚举在领域层定义（`domain/enums/` 包）
- 前后端交互、后端与数据库交互统一以 **Integer code** 为准
- 推荐直接使用枚举类型参数，框架自动完成 Integer ↔ Enum 转换

**枚举定义**：
```java
// domain/enums/AdminUserStatus.java
import com.cartisan.core.domain.BaseEnum;
import com.cartisan.data.jpa.converter.BaseEnumConverter;
import jakarta.persistence.Converter;

public enum AdminUserStatus implements BaseEnum<AdminUserStatus> {
    ACTIVE(1, "激活"),
    DISABLED(0, "禁用");

    private final Integer code;
    private final String name;

    AdminUserStatus(Integer code, String name) {
        this.code = code;
        this.name = name;
    }

    @Override
    public Integer getCode() {
        return code;
    }

    @Override
    public String getName() {
        return name;
    }

    /**
     * JPA Converter - 必须声明为 public static class
     * <p>
     * 通过 @Converter(autoApply = true) 让 Hibernate 自动应用到所有
     * AdminUserStatus 类型的字段，无需在实体类中显式标注。
     * <p>
     * <b>重要提示：</b>内部类名使用 {@code JpaConverter} 而非 {@code Converter}，
     * 避免与 {@code @Converter} 注解产生命名冲突。
     */
    @Converter(autoApply = true)
    public static class JpaConverter extends BaseEnumConverter<AdminUserStatus> {
        public JpaConverter() {
            super(AdminUserStatus.class);
        }
    }
}
```

**在聚合根中使用**：
```java
// 无需 @Convert 或 @Enumerated 注解
// AdminUserStatus 内部的 JpaConverter 会自动应用
@Column(name = "status", nullable = false)
private AdminUserStatus status;
```

**架构边界规则**：

**推荐做法**：直接使用枚举类型参数，框架自动完成 Integer ↔ Enum 转换

| 层级 | 处理方式 |
|------|---------|
| **Controller** | 直接使用枚举类型参数（@RequestParam、@RequestBody 字段、@PathVariable）|
| **AppService** | 直接使用枚举类型参数 |
| **Domain** | 定义枚举（独立文件） |
| **数据库** | 通过内部 `JpaConverter` 自动转换为 Integer code 存储 |

**示例**：
```java
// Controller - 直接使用枚举类型
@Operation(summary = "修改管理员状态")
public void updateStatus(
        @PathVariable Long id,
        @RequestParam AdminUserStatus status) {  // Spring MVC 自动转换 Integer → AdminUserStatus
    adminManagementAppService.updateStatus(id, status);
}

// Response DTO - 直接使用枚举类型
public record AdminUserResponse(
    Long id,
    AdminUserStatus status,      // Jackson 自动序列化为 Integer code
    String statusName            // MapStruct 自动调用 getName()
) {}
```

**BaseEnum 接口方法说明**：
- **`getCode()`**：返回 Integer 类型的编码
- **`getName()`**：返回 String 类型的显示名称
- **`requireByCode(Class, Integer)`**：通过 code 查找枚举（找不到抛异常）
- **`parseByCode(Class, Integer)`**：通过 code 查找枚举（找不到返回 null）

**禁止**：
1. **使用 String 传递枚举值**
2. **前端使用枚举名称**（如 `{ "status": "ACTIVE" }`）
3. **前端使用整型 code**（如 `{ "status": 1 }`）✅

#### 3.6.2 常量规范

**原则**：不涉及前后端或数据库转换

**定义位置**：常量在聚合根中定义为 `public static final`

```java
@Aggregate
public class AdminRole extends AuditableSoftDeletable implements AggregateRoot<AdminRole> {

     */

        return SUPER_ADMIN_CODE.equals(this.code);
    }
}
```

**常量类型示例**：
```java
// String 类型的常量
public static final String STATUS_PENDING = "PENDING";

// Integer 类型的常量
public static final Integer MIN_ORDER_AMOUNT = 100;

// Long 类型的常量
```

**常量命名规范**：全大写，下划线分隔，添加 JavaDoc 注释说明用途

## 四、应用层

### 4.1 应用服务（AppService）

**位置**：`application`  
**命名规范**：`{Scenario}AppService.java`

**实际示例**（Admin 上下文）：
```
application/
├── AdminUserManagementAppService.java      # 管理员管理（CRUD）
├── AdminUserAuthAppService.java            # 管理员认证（登录、注销）
├── AdminUserPermissionAppService.java      # 管理员权限查询
├── MenuManagementAppService.java           # 菜单管理
├── RoleManagementAppService.java           # 角色管理
└── PermissionScanAppService.java           # 权限扫描
```

**命名原则**：
- 按业务职责命名
- 一个上下文可以有多个 AppService
- 体现具体业务场景（Management、Auth、Query 等）

**架构原则**：AppService 不应暴露领域模型给外部

| 返回类型 | 说明 |
|---------|------|
| DTO（Response） | 对外提供的数据 |
| 基本类型（Long/void/boolean） | 操作结果 |
| **领域模型** | ❌ 禁止对外暴露 |

```java
@Service
public class AdminUserManagementAppService {

    private final AdminUserRepository adminUserRepository;
    private final AdminUserMapper adminUserMapper;

    public AdminUserManagementAppService(
            AdminUserRepository adminUserRepository,
            AdminUserMapper adminUserMapper) {
        this.adminUserRepository = adminUserRepository;
        this.adminUserMapper = adminUserMapper;
    }

    /**
     * 查询管理员列表（分页）。
     */
    @Transactional(readOnly = true)
    public PageResponse<AdminUserResponse> findAll(AdminUserQuery query, Pageable pageable) {
        Specification<AdminUser> spec = ConditionSpecifications.fromAnnotation(query);
        Page<AdminUser> page = adminUserRepository.findAll(spec, pageable);

        return new PageResponse<>(
                adminUserMapper.convertList(page.getContent()),
                page.getTotalElements(),
                pageable.getPageNumber() + 1,
                pageable.getPageSize()
        );
    }

    /**
     * 查询管理员详情。
     */
    @Transactional(readOnly = true)
    public AdminUserResponse findById(Long id) {
        AdminUser adminUser = adminUserRepository.findById(id)
                .orElseThrow(() -> new ApplicationException(AdminMessage.ADMIN_NOT_FOUND));

        return adminUserMapper.convert(adminUser);
    }

    /**
     * 创建管理员。
     */
    @Transactional
    public Long create(CreateAdminUserCommand command) {
        require(
                !adminUserRepository.existsByUsername(command.username()),
                AdminMessage.USERNAME_ALREADY_EXISTS
        );

        AdminUser adminUser = new AdminUser(command.username(), command.password(), command.nickname());
        AdminUser saved = adminUserRepository.save(adminUser);
        return saved.getId();
    }
}
```

**应用服务规范要点**：

1. 使用 `@Service` 注解
2. 构造函数注入（禁止 `@Autowired` 字段注入）
3. 读操作用 `@Transactional(readOnly = true)`
4. 写操作用 `@Transactional`
5. **禁止暴露领域模型**：返回类型只能是 DTO、基本类型（Long/void/boolean）
6. 复杂领域逻辑考虑创建 DomainService（按需，非必须）

### 4.3 Mapper 规范

**位置**：`application/mapper`  
**命名规范**：`{Resource}Mapper.java`

```java
@Mapper(componentModel = "spring")
public interface AdminUserMapper extends DomainMapper<AdminUser, AdminUserResponse> {
    // BaseEnum 字段自动映射（如 status → status）
    // xxxName 字段需显式配置（如 statusName ← status.getName()）
    @Mapping(target = "statusName", source = "status.name")
    @Override
    AdminUserResponse convert(AdminUser adminUser);

    // convertList() 和 convertSet() 由 DomainMapper 基类提供
}
```

**Mapper 配置要点**：

**使用方式**：

```java
List<AdminUserResponse> responses = adminUserMapper.convertList(adminUsers);
```

### 5.1 适配器实现规范

**南向接口适配器按技术拆分**：

**示例 1**：密码编码适配器（单个适配器放根目录）
```java
@Adapter(PortType.CLIENT)
public class BCryptPasswordEncoderAdapter implements PasswordEncoder {

    private static final int STRENGTH = 10;
    private final BCryptPasswordEncoder encoder;

    public BCryptPasswordEncoderAdapter() {
        this.encoder = new BCryptPasswordEncoder(STRENGTH);
    }

    @Override
    public String encode(String plainPassword) {
        return encoder.encode(plainPassword);
    }

    @Override
    public boolean matches(String plainPassword, String encodedPassword) {
        return encoder.matches(plainPassword, encodedPassword);
    }
}
```

**示例 2**：跨上下文调用适配器（多个适配器时按技术拆分）
```java
// infrastructure/AdminUserClient.java（接口定义）
public interface AdminUserClient {
    boolean isAdminExists(Long userId);
}

// infrastructure/LocalAdminUserClientAdapter.java（适配器实现）
@Component
public class LocalAdminUserClientAdapter implements AdminUserClient {
    private final AdminUserQueryAppService adminUserQueryAppService;

    @Override
    public boolean isAdminExists(Long userId) {
        return adminUserQueryAppService.existsById(userId);
    }
}
```

**适配器规范要点**：
1. 使用 `@Adapter` 注解标记
2. 实现领域层定义的端口接口
3. 构造函数注入依赖
4. 适配器测试无需 Mock（直接测试实现）

### 5.2 JPA Repository 实现

**位置**：`infrastructure/persistence`

```java
@Repository
public class SpringDataJpaAdminUserRepository implements AdminUserRepository {

    private final JpaSpecificationExecutor<AdminUser> jpaSpecificationExecutor;

    @Override
    public Optional<AdminUser> findByUsername(String username) {
        return jpaSpecificationExecutor.findOne(
            (root, query, cb) -> cb.equal(root.get("username"), username)
        );
    }

    @Override
    public boolean existsByUsername(String username) {
        return jpaSpecificationExecutor.exists(
            (root, query, cb) -> cb.equal(root.get("username"), username)
        );
    }
}
```

**注意**：Spring Data JPA 的 Repository 实现通常可以省略，Spring 会自动生成代理类。只有在需要自定义查询方法时才需要手动实现。

## 六、北向接口层

### 6.1 内部 REST API Controller

**位置**：`endpoints/controller`  
**命名规范**：`{Resource}Controller.java`  
**用途**：供内部前端（Next.js、Vue 等）调用

**URL 规范**：`/api/{context}/{resources}`（无版本号）

```java
@RestController
@RequestMapping("/api/admin/users")
@Validated
@Tag(name = "Admin Users", description = "管理员管理")
public class AdminUserController {

    private final AdminUserManagementAppService adminManagementAppService;

    public AdminUserController(AdminUserManagementAppService adminManagementAppService) {
        this.adminManagementAppService = adminManagementAppService;
    }

    @GetMapping
    @RequireAuth
    @RequirePermission(
        value = "admin:user:read",
        name = "平台管理 / 用户管理 / 查看",
        scope = "admin"
    )
    @Operation(summary = "查询管理员列表（分页）")
    public PageResponse<AdminUserResponse> findAll(
            AdminUserQuery query,
            @PageableDefault(size = 20) Pageable pageable) {
        return adminManagementAppService.findAll(query, pageable);
    }

    @PostMapping
    @RequireAuth
    @RequirePermission(
        value = "admin:user:write",
        name = "平台管理 / 用户管理 / 编辑",
        scope = "admin"
    )
    @Operation(summary = "创建管理员")
    public Long create(@Valid @RequestBody CreateAdminUserCommand command) {
        return adminManagementAppService.createAndReturnId(command);
    }
}
```

**控制器规范要点**：

1. 构造函数注入
2. 类级别加 `@Validated`
3. 权限注解：
   - `@RequireAuth`：需要登录
   - `@RequirePermission`：需要指定权限（`{context}:{module}:{action}` 格式）
4. 参数校验使用 `@Valid`
5. Swagger 注解：`@Tag`（类级别）、`@Operation`（方法级别）
6. 返回类型：`PageResponse<T>` 或具体 DTO/Long/void
7. URL 规范：`/api/{context}/{resources}`（无版本号，内部接口）

### 6.2 外部 API Controller（第三方调用）

**位置**：`endpoints/api/v1`  
**命名规范**：`{Resource}ApiV1Controller.java`

> **重要**：类名必须包含版本号（如 `V1`），避免多版本共存时 Spring Bean 名称冲突。

**用途**：供第三方系统、移动端、开放平台调用

**URL 规范**：`/api/v1/external/{context}/{resources}`

**与前端 Controller 的差异**：

| 维度 | 前端 Controller | 外部 API Controller |
|------|-----------------|---------------------|
| **位置** | `endpoints/controller` | `endpoints/api/v1` |
| **命名** | `{Resource}Controller` | `{Resource}ApiV1Controller` |
| **URL 路径** | `/api/{context}/{resources}` | `/api/v1/external/{context}/{resources}` |
| **认证方式** | Session/JWT | API Key/OAuth2.0 |
| **权限控制** | `@RequirePermission` | API Key 权限 |
| **限流策略** | 用户级 | API Key 级 |
| **版本控制** | 相对宽松 | 严格版本管理（v1/v2） |
| **错误响应** | 可返回详细错误 | 标准化错误码 |
| **文档要求** | 内部文档 | 公开 API 文档 |

```java
@RestController
@RequestMapping("/api/v1/external/admin/admin-users")
@Validated
@Tag(name = "External Admin Users API v1", description = "外部管理员管理 API v1")
public class AdminUserApiV1Controller {

    private final ApiKeyAuthenticator apiKeyAuthenticator;
    private final AdminUserManagementAppService adminManagementAppService;

    public AdminUserApiV1Controller(
            ApiKeyAuthenticator apiKeyAuthenticator,
            AdminUserManagementAppService adminManagementAppService) {
        this.apiKeyAuthenticator = apiKeyAuthenticator;
        this.adminManagementAppService = adminManagementAppService;
    }

    @GetMapping("/{id}")
    @ApiKeyAuth  // API Key 认证
    @RateLimit(rate = 100, interval = 60)  // 限制：每分钟 100 次
    @Operation(summary = "查询管理员详情（外部 API v1）")
    public ApiResponse<AdminUserResponse> getById(@PathVariable Long id) {
        AdminUserResponse response = adminManagementAppService.findById(id);
        return ApiResponse.success(response);
    }
}
```

**外部 API Controller 规范要点**：

1. **类名包含版本号**：`{Resource}ApiV1Controller`，避免 Bean 名称冲突
2. 使用 API Key 认证（或 OAuth2.0）
3. 添加严格的限流注解（按 API Key）
4. 返回统一的响应格式（`ApiResponse<T>`）
5. 错误码标准化，不暴露内部实现细节
6. URL 中包含 `external` 前缀，标识为外部 API
7. 严格的版本控制（v1、v2 共存）
8. 完善的 API 文档（Swagger/OpenAPI）

### 6.3 消息队列监听器（MQ Listener）

**位置**：`endpoints/listener`  
**命名规范**：`{Event}Listener.java` 或 `{Event}MessageListener.java`

**用途**：监听消息队列消息，接收后调用应用服务处理

**与 Controller 的关系**：

| 维度 | REST Controller | MQ Listener |
|------|-----------------|-------------|
| **触发方式** | HTTP 请求 | 消息队列消息 |
| **同步/异步** | 同步（阻塞） | 异步（非阻塞） |
| **返回值** | 立即返回响应 | 确认消息接收 |
| **错误处理** | 返回错误响应 | 死信队列或重试 |
| **典型场景** | 前端交互、外部 API | 异步任务、事件通知 |

```java
@Component
public class UserCreatedListener {

    private final AdminUserManagementAppService adminUserManagementAppService;

    /**
     * 监听用户创建事件，自动创建管理员账号。
     */
    @RabbitListener(queues = "user.created")
    public void handleUserCreated(UserCreatedEvent event) {
        try {
            adminUserManagementAppService.createAdminForUser(event.getUserId(), event.getUsername());
        } catch (Exception e) {
            log.error("Failed to handle user created event: {}", event, e);
            throw e;  // 抛出异常让 MQ 框架处理重试
        }
    }
}
```

**MQ Listener 规范要点**：

1. **包位置**：`endpoints/listener`
2. **命名规范**：`{Event}Listener.java`
3. **注解**：使用 `@Component`
4. **构造函数注入**：注入需要的 AppService
5. **异常处理**：记录详细日志，抛出异常让 MQ 框架处理重试
6. **幂等性**：确保消息重复处理时不会产生副作用
7. **事务管理**：Listener 方法通常不加 `@Transactional`，AppService 方法内部加

## 七、数据库规范

### 7.1 表命名规范

**格式**：`{prefix}_{table_name}`

- **prefix**：前缀，目的是让同一上下文的表在数据库中聚集在一起，便于管理
  - 可使用上下文缩写（如 `adm`、`act`、`tnt`）
  - 也可使用其他有意义的前缀（如 `sys` 表示系统表）
  - **关键原则**：同一上下文的所有表使用相同前缀
- **table_name**：蛇形命名（snake_case）

| 上下文 | 前缀 | 示例 |
|--------|------|------|
| admin | `sys` | `sys_admin_users`, `sys_admin_roles`, `sys_admin_menus` |
| account | `act` | `act_users`, `act_profiles` |
| tenant | `tnt` | `tnt_tenants`, `tnt_members` |
| course | `crs` | `crs_courses`, `crs_lessons` |
| platform | `plt` | `plt_users`, `plt_configs` |

**前缀选择示例**：
- `sys_admin_users`：使用 `sys` 前缀，表示系统级管理员表
- `adm_users`：使用上下文缩写 `adm`，更简洁
- **两种方式都正确**，选择时考虑团队习惯和表的可读性

### 7.2 迁移脚本规范

**位置**：`src/main/resources/db/migration`  
**命名规范**：`V{version}__{description}.sql`

```sql
-- ========================================================================
-- {Context Name} Context: {Description}
-- ========================================================================

-- 主表
CREATE TABLE {prefix}_{table_name} (
    id              BIGINT PRIMARY KEY,
    name            VARCHAR(50) NOT NULL,
    -- 其他字段...
    created_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    created_by      BIGINT,
    updated_by      BIGINT,
    deleted         BOOLEAN NOT NULL DEFAULT FALSE
);

-- 关联表
CREATE TABLE {prefix}_{relation_table} (
    parent_id       BIGINT NOT NULL,
    child_id        BIGINT NOT NULL,
    PRIMARY KEY (parent_id, child_id)
);

-- 索引（软删除过滤）
CREATE INDEX idx_{prefix}_{table}_{column} ON {prefix}_{table_name}({column}) WHERE deleted = FALSE;

-- 外键
ALTER TABLE {prefix}_{relation_table} ADD CONSTRAINT fk_{prefix}_{relation}_{parent}
    FOREIGN KEY (parent_id) REFERENCES {prefix}_{parent_table}(id);
```

**标准字段**：
- `id`：BIGINT 主键（TSID）
- `created_at`、`updated_at`：审计时间
- `created_by`、`updated_by`：审计人（BIGINT）
- `deleted`：软删除标记（BOOLEAN，默认 FALSE）

### 8.1 单元测试

**命名规范**：`{ClassName}Test.java`  
**测试方法命名**：`given_{条件}_when_{操作}_then_{预期结果}`

```java
class AdminUserTest {

    @Test

        assertThat(adminUser.getStatus()).isEqualTo(AdminUserStatus.ACTIVE);
    }

    @Test
    void given_invalid_username_when_create_admin_then_throw_exception() {
        // When & Then

                .hasMessageContaining(AdminMessage.USERNAME_INVALID.message());
```

**断言规范**：
1. 使用 AssertJ（禁止 JUnit 断言）
2. 异常测试使用 `assertThatThrownBy`

**单元测试覆盖范围**：

| 领域服务测试 | `domain/service` | 测试领域服务（使用 Mockito） | ✅ 必须 |

### 9.3 工具库使用规范（hutool）

#### 9.3.1 集合初始化

**使用 hutool 替代 Java 标准库**：

| Java 标准库 | hutool 替代 |
|------------|-------------|
| `new HashSet<>()` | `CollUtil.newHashSet()` |
| `new ArrayList<>()` | `CollUtil.newArrayList()` |
| `new HashMap<>()` | `MapUtil.newHashMap()` |

```java
import cn.hutool.core.collection.CollUtil;
import cn.hutool.core.map.MapUtil;

// 推荐
Set<Long> roleIds = CollUtil.newHashSet();
List<AdminMenu> roots = CollUtil.newArrayList();
Map<Long, AdminMenu> menuMap = MapUtil.newHashMap();

// 不推荐
Set<Long> roleIds = new HashSet<>();
List<AdminMenu> roots = new ArrayList<>();
Map<Long, AdminMenu> menuMap = new HashMap<>();
```

#### 9.3.2 Stream 收集操作

**Stream 操作仍使用 Java 标准库 Collectors**（hutool 不支持 Stream）：

```java
import java.util.stream.Collectors;

// Stream to List
List<String> names = users.stream()
    .map(User::getName)
    .collect(Collectors.toList());

// Stream to Set
Set<Long> ids = roles.stream()
    .map(AdminRole::getId)
    .collect(Collectors.toSet());
```

#### 9.3.3 Java 语言特性保持不变

以下情况继续使用 Java 标准库：

| 场景 | 用法 | 说明 |
|------|------|------|
| 空不可变集合 | `List.of()` | Java 9+，性能最优 |
| Stream 终止操作 | `.toList()` | Java 16+，比 `Collectors.toList()` 更高效 |
| Optional | `Optional.ofNullable()` | Java 标准库 |
| Objects 工具 | `Objects.equals()` | Java 标准库 |

```java
// 推荐（Java 语言特性）
return List.of();  // 空不可变列表
return stream.toList();  // Java 16+ Stream 收集
return Optional.ofNullable(value);  // Optional 包装
if (Objects.equals(a, b)) { }  // null 安全比较
```

## 十、架构守护

### 10.1 ArchUnit 规则

项目级架构测试位于 `src/test/java/com/aieducenter/ArchitectureTest.java`，确保：

1. **分层规则**：领域层不依赖基础设施层
2. **命名规则**：Controller/AppService/Repository 后缀规范
3. **禁止规则**：
   - 禁止字段注入（`@Autowired`）
   - 禁止使用 `java.util.Date`
   - 禁止金额字段使用 float/double

### 10.2 检查清单

新建上下文时，确保：

**基础结构**
- [ ] `package-info.java` 说明文件（含 @BoundedContext 注解）
- [ ] 聚合根**建议**继承 `AuditableSoftDeletable`（使用 `@Getter`/`@Setter` 注解）
- [ ] 实体**建议**不继承 `AuditableSoftDeletable`（避免唯一索引冲突）
- [ ] 表名使用上下文前缀（让同上下文的表在数据库中聚集）
- [ ] 错误定义实现 `CodeMessage` 接口

**应用层**
- [ ] 应用服务使用构造函数注入
- [ ] AppService 不暴露领域模型，只返回 DTO/基本类型
- [ ] Controller **不导入**领域模型（如 `import ...domain.aggregate.AdminUser`）
- [ ] Controller 使用 `@RequirePermission` 注解
- [ ] DTO 使用 Record + 校验注解
- [ ] Mapper 继承 `DomainMapper`，配置 `xxxName` 字段映射

**领域层**
- [ ] 枚举定义在独立文件中（`domain/enums/`），实现 `BaseEnum<T>` 接口
- [ ] 常量定义为 `public static final` 在聚合根中
- [ ] 字符串比较使用常量，禁止硬编码（如 `"SUPER_ADMIN"` → `SUPER_ADMIN_CODE`）

**基础设施层**
- [ ] 适配器按南向接口技术拆分子包（persistence、redis、messaging 等）
- [ ] 适配器实现使用 `@Adapter` 注解
- [ ] 适配器测试无需 Mock（直接测试实现）

**代码风格**
- [ ] 测试方法命名遵循 `given_when_then` 规范
- [ ] 导入顺序符合规范（Java → hutool → Jakarta/Spring → cartisan → 项目内部）
- [ ] 集合初始化使用 hutool（`CollUtil.newHashSet()`、`CollUtil.newArrayList()`、`MapUtil.newHashMap()`）
- [ ] Stream 收集使用 Java 标准库（`Collectors.toList()`、`Collectors.toSet()`）
